# Trabalho – Engenharia de Software  
**Entrega final: 10/12**  
**Formato da entrega:** Documento + Diagramas + Apresentação

---

## 1. Documento a ser entregue

O documento deve apresentar de forma clara e organizada os elementos levantados ao longo do projeto.

### 1.1. Requisitos levantados pelo questionário
- Listar os requisitos funcionais e não funcionais identificados.  
- Explicar como cada requisito foi derivado das respostas do questionário.  
- Indicar quais requisitos foram priorizados e os motivos.

### 1.2. Como os requisitos influenciaram a definição do projeto
- Descrever como os requisitos direcionaram o escopo.  
- Destacar funcionalidades essenciais para o MVP.  
- Relacionar necessidades dos usuários com decisões de fluxo e comportamento do sistema.

---

## 2. Diagramas

Os três diagramas abaixo devem ser apresentados, já revisados e corrigidos:

### 2.1. Diagrama de Caso de Uso
- Representar atores e suas interações com o sistema.  
- Destacar casos de uso principais.

### 2.2. Diagrama de Sequência
- Ilustrar o fluxo de mensagens entre objetos/entidades em um cenário específico.  
- Mostrar a ordem dos eventos e suas dependências.

### 2.3. Diagrama de Classes
- Incluir classes, atributos, métodos e relacionamentos.  
- Representar corretamente associações, composições e heranças.  
- Refletir o modelo conceitual atualizado.

---

## 3. Importância dos Diagramas na Prototipação Inicial

### 3.1. Caso de Uso
- Ajuda a visualizar expectativas dos usuários.  
- Define o escopo funcional de forma clara.

### 3.2. Diagrama de Sequência
- Permite validar cenários antes da implementação.  
- Facilita a identificação de falhas no fluxo.

### 3.3. Diagrama de Classes
- Base estrutural para a implementação.  
- Organiza responsabilidades e relacionamentos do sistema.

---

## 4. Arquitetura do Sistema

Apresentar a arquitetura escolhida e justificar brevemente.

### 4.1. Monolito
- Simples de implementar e manter.  
- Adequado para MVPs e sistemas pequenos.

### 4.2. Microserviços
- Melhor escalabilidade e separação de responsabilidades.  
- Requer maior complexidade de comunicação e infraestrutura.

### 4.3. Serverless (Sem Servidor)
- Alta elasticidade e baixo custo operacional.  
- Depende de serviços gerenciados e boa definição de integrações.

A arquitetura selecionada deve refletir os requisitos e o escopo definidos anteriormente.

---

## 5. Apresentação (3–5 minutos)

A apresentação deve incluir:

1. Resumo dos requisitos levantados  
2. Como os requisitos influenciaram o projeto  
3. Apresentação dos diagramas  
4. Explicação da importância dos diagramas  
5. Arquitetura escolhida e justificativa  

---

## Resumo Geral da Entrega

- **Documento:** requisitos + justificativas + importância dos diagramas + arquitetura.  
- **Diagramas:** caso de uso, sequência e classes corrigido.  
- **Apresentação:** vídeo de 3–5 minutos.  
- **Prazo:** 10/12.
